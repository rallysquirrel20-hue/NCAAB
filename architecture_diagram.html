NCAAB Pipeline Architecture

Season 2025-26 · 68 Tournament Teams + 348 Opponents · 3,950 Rows · 127 Columns

1. File Map

ncaab_config.py
Constants: 68 TEAMS list, CONFERENCE_MAP, ESPN_NAME_ALIASES, ODDS_NAME_MAP, HARDCODED_ESPN_IDS, API endpoints, rate limits
ncaab_daily_builder.py
Main pipeline. 4 phases + 5 backfill modes. All data fetching, computation, and export lives here (1,862 lines)
build_team_name_map.py
One-time utility. Samples Odds API on 18 busy dates, fuzzy-matches team names against ESPN directory → team_name_map.json
.env
THE_ODDS_API_KEY (secret)
espn_team_ids.json
Cache: canonical team name → ESPN numeric ID for 68 tournament teams
team_name_map.json
430-entry map: ESPN display name → Odds API long-form name (e.g. "Arizona Wildcats" → "Arizona Wildcats")
ncaab_game_logs.parquet
Primary data store. 127 columns × 3,950 rows. Read at start, written after Phase 2 and Phase 4
ncaab_game_logs.csv
Export for analysis/Excel. 102 columns × 2,272 rows (tracked teams only). Written by Phase 4

Phase 3 PIT_COLUMNS (93 columns) — Computed, Zero API

All PIT stats are Point-In-Time: they reflect the team's performance PRIOR to each game (not including that game). Computed for all 416 teams in the dataframe, enabling opponent PIT lookup for every row.

5. CLI Entry Points

CommandWhat It DoesAPI Calls
python ncaab_daily_builder.pyDaily run: fetch new games, daily odds (1 snapshot/date), mirror rows, recompute PIT, export CSV1 scoreboard + 1 summary per new game + 1 odds per new date
--backfill-oddsTrue opening/closing lines: 1 snapshot per unique commence hour~718 Odds API calls (~14,360 credits)
--backfill-opponentsFetch full schedules for non-tournament opponents + boxscores~348 schedule + ~32 summary (ESPN, free)
--backfill-boxscoreRe-fetch summary for rows missing boxscore stats1 ESPN call per event missing data
--backfill-ranksRe-scan scoreboards to fill AP ranks1 ESPN call per date with missing ranks
--backfill-timesBackfill commence_time_utc for existing rows1 ESPN call per date with missing times

6. Data Flow: How a Single Game Becomes a Row

sequenceDiagram
  participant M as main()
  participant ESPN as ESPN API
  participant ODDS as Odds API
  participant PQ as Parquet
  participant CSV as CSV

  Note over M: Phase 1
  M->>ESPN: GET /scoreboard?dates=20260319
  ESPN-->>M: events[] (all games that day)
  M->>M: parse_scoreboard() → filter to tracked teams
  M->>ESPN: GET /summary?event=401234567
  ESPN-->>M: boxscore: FGM, FGA, 3PM, 3PA, FTM, FTA, OREB, DREB, TO
  M->>M: Append row with 30 RAW columns

  Note over M: Phase 2 (daily)
  M->>ODDS: GET /historical/odds?date=2026-03-18T22:00:00Z
  ODDS-->>M: All NCAAB lines at that moment
  M->>M: Match team via _TEAM_NAME_MAP
  M->>M: Extract h2h→ML, spreads→point from first bookmaker
  M->>M: Set opening = closing (single snapshot)

  Note over M: Mirror Rows
  M->>M: For non-tracked opponents: swap team↔opp columns
  M->>M: is_tracked = False

  Note over M: Save checkpoint
  M->>PQ: save_game_logs() → parquet (type coercion + orphan cleanup)

  Note over M: Phase 3 — PIT (pure math)
  M->>M: For each of 416 teams: walk games chronologically
  M->>M: Accumulate W/L, points, shooting, rebounds, TOs
  M->>M: SAVE stats BEFORE each game → entry dict
  M->>M: After loop: write team_* and opp_* columns

  Note over M: Phase 4
  M->>M: Filter to is_tracked == True (2,272 rows)
  M->>M: Select 102 columns, rename headers
  M->>CSV: Write ncaab_game_logs.csv
  M->>PQ: Final save with PIT columns

7. Where Predictions Would Plug In

No prediction model exists yet. The pipeline produces features (PIT stats, odds, boxscore) but no target variable or model. A prediction phase would slot in after Phase 3 (features computed) and before/alongside Phase 4 (export).
